# Documentation Rules

## 1. 문서는 역할 기준으로 분류한다

파일 이름이나 기능명보다 문서가 독자에게 하는 역할을 기준으로 둔다.

분류 기준:

- `structure/`: 현재 프로젝트 구조, 데이터 모델, 화면, 런타임 흐름, AI 계약
- `rules/`: 개발과 설계에서 지켜야 할 원칙과 그 이유
- `scenarios/`: 실제 플레이 시나리오 본문
- `examples/`: 사용 예시, JSON 예시, 운영 팁, 테스트 체크리스트
- `completed/`: 완료된 구현 계획, 과거 초안, 더 이상 현재 작업 체크리스트가 아닌 문서
- `dev-notes/`: 특정 날짜의 구현 기록과 당시 판단

이유:

문서가 기능별로만 쌓이면 새로 합류한 사람이 “현재 기준”, “과거 기록”, “예시”, “원칙”을 구분하기 어렵다.

## 2. 루트에는 현재 진행 문서만 둔다

`doc` 바로 아래에는 전체 문서 지도와 현재 진행 중인 작업을 둔다.

루트 유지 대상:

- `README.md`
- `PENDING_WORK_ITEMS.md`
- `future_plan.md`
- `future_plan_p3.md`

이유:

루트가 깨끗해야 지금 봐야 할 문서와 참고용 문서를 빠르게 구분할 수 있다.

## 3. 중복은 링크로 관리한다

같은 내용을 여러 문서에 자세히 반복하지 않는다.

지켜야 할 것:

- 제품 범위는 `structure/PRODUCT_SCOPE.md`를 기준으로 한다.
- 완료 기준은 `structure/QUALITY_MVP_ACCEPTANCE.md`를 기준으로 한다.
- 데이터 모델은 `structure/ERD_MVP_SESSION_SERVICE_MODEL.md`를 기준으로 한다.
- AI 계약은 `structure/AI_RUNTIME_CONTRACTS.md`를 기준으로 한다.
- 설계 원칙은 `rules/` 문서로 모은다.
- 다른 문서에서는 요약과 링크만 둔다.

이유:

중복 문서가 많아지면 한쪽만 최신화되어 충돌이 생긴다. 기준 문서 하나를 두고 나머지는 링크로 연결해야 유지 비용이 낮다.

## 4. 완료된 계획은 현재 작업 목록에서 뺀다

이미 구현이 끝난 계획 문서는 `completed/`로 이동한다.

지켜야 할 것:

- 완료된 구현 계획, UI 설계안, 점검표는 루트나 `structure/`에 남기지 않는다.
- 완료 문서에서 아직 남은 작업이 있으면 `PENDING_WORK_ITEMS.md`로 따로 추출한다.
- 완료 문서는 히스토리로 보되 현재 기준보다 우선하지 않는다.

이유:

완료된 계획 문서가 루트에 남으면 아직 해야 할 작업처럼 보인다. 남은 일과 완료 기록을 분리해야 작업 우선순위가 명확해진다.

## 5. 실제 코드와 충돌하면 문서를 갱신한다

문서는 현재 구현을 이해하기 위한 도구다.

지켜야 할 것:

- 구조 문서가 현재 Prisma, API, DTO, 화면 조건과 충돌하면 문서를 최신화한다.
- 과거 판단을 보존해야 하면 `completed/` 또는 히스토리 섹션으로 내린다.
- 새 원칙이 생기면 `rules/`에 추가하고 관련 구조 문서에서 링크한다.

이유:

낡은 문서는 없는 문서보다 더 위험할 수 있다. 특히 권한, AI 상태 변경, 라이선스 같은 항목은 오래된 설명이 곧 버그나 위험으로 이어진다.
