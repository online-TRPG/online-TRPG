## 1.1 시스템 아키텍처

```
                          ┌─────────────────────────────────────────────┐
   사용자 ── HTTPS ──▶     │  Nginx (80/443)                              │
                          │   - React 정적 파일 서빙                      │
                          │   - /api  → backend:8080 리버스 프록시        │
                          │   - WebSocket(Socket.IO) 프록시               │
                          └───────┬───────────────────────┬──────────────┘
                                  │                       │
                       ┌──────────▼─────────┐   ┌─────────▼──────────┐
                       │ backend (NestJS)   │   │ ai-server (FastAPI)│
                       │  :8080             │──▶│  :8000             │──▶ Google AI Studio
                       └──┬──────────────┬──┘   └────────────────────┘
                          │              │
                ┌─────────▼───┐   ┌──────▼──────┐
                │ PostgreSQL  │   │ Redis       │
                │  :5432      │   │  :6379      │
                └─────────────┘   └─────────────┘

   * 모든 컨테이너는 EC2 단일 호스트에서 Docker Compose 로 구동됩니다.
   * postgres / redis / ai-server / backend 는 외부 포트 비노출, Nginx 만 80/443 노출.
```

## 1.4 환경 변수

`.env.example` 을 템플릿으로, 레포 루트에 아래 **3개 파일**을 생성하고 값을 채웁니다.
세 파일 모두 `.gitignore` 처리되어 있으며, 운영 환경에서는 Jenkins Credentials(Secret file)로 관리됩니다.

> ⚠️ 실제 키/비밀번호 값은 보안상 본 문서에 기재하지 않습니다. 운영 값은 Jenkins Credentials 또는 팀 보안 채널을 통해 전달받으세요.

## 1.5 빌드 및 배포 절차

### 방법 A. 수동 배포 (EC2 호스트에서 직접)

```bash
# 1) 소스 클론
git clone https://lab.ssafy.com/s14-final/S14P31A201.git
cd S14P31A201

# 2) 환경변수 파일 3종 생성 (.env.example 참고)
#    .env / .env.backend / .env.ai 를 레포 루트에 작성

# 3) 이미지 빌드 (FE 빌드는 nginx 멀티스테이지 Dockerfile 내부에서 자동 수행)
docker compose build --pull

# 4) 데이터 계층 먼저 기동
docker compose up -d --wait postgres redis ai-server

# 5) DB 스키마 동기화 (Prisma db push — 마이그레이션 파일 미사용)
docker compose run --rm --entrypoint "" backend \
  sh -c "cd /app/be && npx prisma db push --schema prisma/schema.prisma --skip-generate"

# 6) 카탈로그/시나리오 시드 (Race·Item·ClassDefinition 등, 모두 upsert 라 멱등)
docker compose run --rm --entrypoint "" backend \
  sh -c "cd /app/be && npx prisma db seed"

# 7) 전체 기동
docker compose up -d
docker compose ps
```

### 방법 B. Jenkins CI/CD (운영 표준)

- **트리거**: GitLab Webhook — `develop` 브랜치 Push 또는 Merge Request 이벤트
- **배포 대상**: `develop` 브랜치만 (`when { branch 'develop' }`)
- **파이프라인 단계** (`Jenkinsfile`):
  1. `Checkout` — SCM 체크아웃
  2. `Prepare env files` — Jenkins Credentials(Secret file) → `.env` / `.env.backend` / `.env.ai` 복사
  3. `Compose: build images` — `docker compose build --pull`
  4. `Deploy (develop only)`:
     - 데이터 계층 기동 → **`pg_dump` 백업** → `prisma db push` → `prisma db seed` → 전체 기동 → **Smoke 테스트**(nginx/ai/backend 헬스 체크)
- **Jenkins Credentials (Secret file)**: `env-root`, `env-backend`, `env-ai`

### 방법 C. 로컬 PC 임시 공유 (Cloudflare Tunnel)

개발자가 자신의 PC에서 현재 구현을 외부 팀원에게 확인시키는 용도다. 운영 배포가 아니며, `trycloudflare.com` 임시 URL은 터널을 새로 열 때마다 바뀐다.

현재 표준은 **프론트 터널 하나만 여는 방식**이다.

```text
https://<random>.trycloudflare.com
  -> cloudflared
  -> Vite dev server http://localhost:5173
     - /api      -> http://localhost:8080
     - /socket.io -> http://localhost:8080
```

실행:

```powershell
npm run start -w @trpg/be
npm run dev -w @trpg/fe
cloudflared tunnel --url http://localhost:5173
```

백엔드용 터널은 별도로 열지 않는다. 프론트 dev server가 REST API와 Socket.IO를 백엔드로 프록시한다.

확인:

```powershell
curl https://<random>.trycloudflare.com/api/v1/health
curl "https://<random>.trycloudflare.com/socket.io/?EIO=4&transport=polling"
```

Kakao/Discord OAuth까지 테스트할 때는 각 콘솔의 redirect URI를 현재 프론트 터널 URL로 갱신한다.

```text
https://<random>.trycloudflare.com/oauth/callback
```

세부 절차와 문제 해결은 `doc/LOCAL_CLOUDFLARE_TUNNEL.md`를 따른다.

## 1.6 배포 시 특이사항

1. **DB 마이그레이션 미사용 — `prisma db push` 방식**
   초기 개발 단계라 마이그레이션 파일 없이 `db push` 로 스키마를 동기화합니다.
   `--accept-data-loss` 는 의도치 않은 데이터 손실 방지를 위해 **제거**되어 있어,
   스키마 변경이 데이터 손실을 동반하면 해당 단계가 실패합니다. 수동 검토 후 진행하세요.

2. **배포 직전 자동 DB 백업**
   매 배포 시 `pg_dump` 로 `db_backups` named volume 에 백업을 남깁니다(최근 20개 유지).
   파일명: `pre-deploy-<branch>-<build#>-<short_sha>.dump` — 자세한 내용은 `3_DB덤프_가이드.md` 참고.

3. **시드는 멱등(idempotent)**
   `prisma db seed` 는 모두 `upsert` 라 여러 번 실행해도 안전합니다.
   운영 이미지에는 `src/`·`tsx` 가 없어 `dist/database/seed.js` 산출물을 호출합니다.

4. **env 파일(Credential) 변경 시 컨테이너 강제 재생성 필요**
   `docker compose up -d` 는 이미지 해시가 같으면 컨테이너를 재생성하지 않습니다.
   Jenkins Credential(`env-*`) 회전 후에는 EC2 에서 수동으로:
   ```bash
   docker compose up -d --force-recreate backend ai-server
   ```

5. **DooD (Docker outside of Docker)**
   Jenkins 컨테이너가 호스트 `docker.sock` 을 마운트해 호스트 Docker 데몬을 직접 제어합니다.

6. **SSL 최초 발급은 1회 수동**
   Let's Encrypt 인증서 최초 발급만 수동, 이후 Certbot 이 12시간 주기로 자동 갱신합니다.
   (명령은 `docker-compose.yml` 의 `certbot` 서비스 주석 참고)

7. **멀티브랜치 — 마지막 커밋 반영 시점**
   Jenkins 멀티브랜치 빌드는 빌드 시작 시점의 `develop` HEAD 를 체크아웃합니다.
   직전 빌드 이후 push 된 커밋은 다음 빌드가 트리거될 때 반영됩니다.
